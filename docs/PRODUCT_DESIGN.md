# Party Level Architect — プロダクト体験設計図 & 本番実装システム仕様

> **コアメッセージ:** 「想像できる事は創造できる。デザインができる皆は、それを言語化できるだけで全てを完成させれる潜在的な才能がある」
>
> **体験の正体:** 美大生クリエイターが「言葉」と「配置」だけで、目の前の大画面に動くゲームを10分で誕生させる。
> **学びの正体:** アートディレクションは、そのままゲームエンジンのパラメータに翻訳できる——その翻訳過程を**全員に見せる**こと。

---

## 0. プロダクト概要（1ページサマリー）

| 項目 | 内容 |
|---|---|
| 形式 | ホストPC（メインスクリーン投影）×1 + 参加者スマホ×2〜3 のリアルタイム協働ゲーム |
| 所要時間 | 10分（接続1.5分 / 創造5分 / ビルド1分 / プレイ1.5分 / 解説1分） |
| 参加者の役割 | A: 環境とテンポ / B: 敵とギミック / C: UIとエフェクト |
| 入力手段 | テキスト（言葉による定義）+ スタンプ配置（ミニマップへのドラッグ） |
| 出力 | Phaser 3 で動く2Dパーティアクションゲーム（横スクロール・ラン&ジャンプ型） |
| 翻訳エンジン | Claude API による「言葉 → StageParams（JSON）」の構造化変換 |
| 設計上の最重要原則 | **翻訳過程を隠さない。** 「ネオン輝くディスコ」が `bpm: 128, palette: "neon", scrollSpeed: 1.4` に変わる瞬間を、ビルドログとして大画面に流す。これが学びの本体。 |

## 1. UX/エモーショナル・ジャーニーマップ

### 1.1 感情曲線の設計方針

10分間で「**違和感 → 渇望 → 没入 → 期待 → 熱狂 → 誇り**」の6段階を駆け抜けさせる。
鍵は2つ:

1. **冒頭の「殺風景」を意図的に長く見せない**こと（渇望は60秒で十分。それ以上は退屈に転じる）
2. **入力→反映のフィードバックを「自分の仕業だ」と確信できる形にする**こと（誰の入力が何を変えたか、必ず名前付きで可視化）

### 1.2 タイムライン詳細

#### ◆ Phase 0: 渇望（0:00 – 1:00）「この世界、まだ何もない」

| 要素 | 設計 |
|---|---|
| PC画面 | ワイヤーフレームの床、グリッド線、無音。中央でダミーキャラ（白い箱人間）が所在なげに足踏み。画面隅に小さく `BUILD: 0% — NO WORLD DATA` |
| 演出アイデア | ダミーキャラが時々こちらを見て肩をすくめる。10秒に1回、画面に「ここには まだ 音がない」「ここには まだ 色がない」と無機質なテロップ |
| 参加者の感情 | 違和感 →「描きたい」「埋めたい」という制作者特有のムズムズ（**渇望**） |
| ホストの一言 | 「この世界には何もありません。あるのは才能を持った皆さんだけです」 |

#### ◆ Phase 1: 接続（1:00 – 1:30）「私はこの世界の建築家になった」

| 要素 | 設計 |
|---|---|
| 接続フロー | PC画面にQRコード表示 → スマホで読む → 名前入力 → 役割カード（A/B/C）が**カードを引く演出**で配布 |
| 心地よいフィードバック | 接続成功の瞬間: ①スマホが短く振動 + 役割カラーで画面がフラッシュ ②PC画面のワイヤーフレーム空間に「**ARCHITECT ONLINE: ○○**」のネオンサインが点灯、起動音（電源が入るような低音→上昇音） |
| 役割カード文言 | A「環境とテンポの設計者 — この世界の空気と心臓の速さは、あなたが決める」/ B「敵とギミックの設計者 — この世界の障害と驚きは、あなたが決める」/ C「UIとエフェクトの設計者 — この世界の手触りと輝きは、あなたが決める」 |
| 参加者の感情 | 自分だけの管轄領域をもらった当事者感。**「建築家」という呼称が自尊心に火を点ける** |

#### ◆ Phase 2: 体現（1:30 – 6:30）「言葉が世界になる」 — 体験の心臓部

入力は「お題カード形式」で1人3問。自由記述+スタンプ配置。

**入力→反映のフィードバック設計（最重要）:**

| ステップ | スマホ側 | PC側（メインスクリーン） | 音 |
|---|---|---|---|
| ① 送信 | 送信ボタンを押すと文字が画面上方へ「飛んでいく」アニメ + 軽い振動 | 画面端から光の粒が飛来し、対象領域の上で渦を巻く | シュッ（発射音） |
| ② 翻訳中 | 「世界が あなたの言葉を 読んでいます…」+ プログレスリング | 渦の中心に入力された言葉がそのまま表示され、文字が**分解されてパラメータに変わっていく**（例: 「ネオン輝くディスコ」→ `palette: neon` `bgm: edm` `bpm: 128` が一行ずつ流れる） | タイプ音（カタカタ） |
| ③ 反映 | 「REFLECTED ✓」スタンプが押される演出 + 強めの振動1回 | 背景色がリップル（波紋）状に塗り替わる / 敵がその場に実体化（ワイヤーフレーム→着色のマテリアライズ演出）/ UIテーマが一斉に切り替わる。**変化した箇所の近くに入力者の名前タグが3秒間表示**（「by ○○」） | ジャンルに応じた決め音（EDMならビルドアップ→ドロップの一瞬） |

> **設計意図:** ②の「翻訳中」表示こそが研修の伏線。遊んでいる間に全員が「言葉がパラメータになる」光景を10回以上目撃する。Phase 4 の解説はこの光景の**種明かし**として機能する。

**お題カードの例（各役割3問・1問あたり制限時間50秒）:**

- **プレイヤーA（環境とテンポ）**
  1. 「この世界の**舞台**はどんな場所？ 見えるものを言葉で描いて」→ 背景・パレット
  2. 「この世界に流れる**音楽**は？ ジャンルと気分を教えて」→ BGMジャンル・BPM
  3. 「このゲームの**スピード感**は？ 例えるなら何の速さ？」→ スクロール速度・重力
- **プレイヤーB（敵とギミック）**
  1. 「行く手を阻む**敵**はどんな姿？ 見た目と性格を描いて」→ 敵スプライト・色・サイズ
  2. 「その敵は**どう動く**？ 跳ねる？追いかける？回る？」→ 挙動パラメータ
  3. 「ステージの**トラップ**を1つ発明して、マップに置いて」→ **スタンプ配置**（ミニマップにドラッグ） |
- **プレイヤーC（UIとエフェクト）**
  1. 「スコアや体力の**表示**はどんなデザイン？ 雰囲気を言葉で」→ UIテーマ・フォント・色
  2. 「ジャンプやアイテム取得の瞬間、**画面に何が舞う**？」→ パーティクル種別・密度
  3. 「クリアの瞬間の**最高の演出**を描いて」→ フィナーレ演出 |

**待ち時間ゼロ設計:** 3人のお題は並行進行。PC画面は3分割せず1つの世界のまま、反映が起きるたびに該当箇所へカメラが軽くパン。「他人の入力で世界が変わる瞬間」も全員が共有し、相互の歓声を誘発する。

| 参加者の感情 | 1問目: 恐る恐る → 反映を見て「えっ、ほんとに変わった！」（**没入**）→ 2問目以降: 言葉選びが大胆に、語彙が増える（クリエイターの本領発揮）→ 仲間の入力への歓声（**共創の快感**） |
|---|---|

#### ◆ Phase 3: ビルド（6:30 – 7:30）「全てが繋がる60秒」

| 要素 | 設計 |
|---|---|
| トリガー | 全入力完了 → PC画面中央に巨大な「**BUILD**」ボタン出現 → **3人のスマホに同時に「同時長押しせよ」の指示**。3人が同時に押し続けると充電ゲージが溜まり、発火（協働の儀式化） |
| ビルド演出 | ①画面の全要素が一旦光の粒子に分解 ②高速で流れる「ビルドログ」: `Translating "跳ね回るスライムドリンク" → bounce: 0.85, moveSpeed: 140, tint: #7CFC00` のような行が**実際の入力文言入り**で20行以上流れる ③プログレスバー 0→100% ④ホワイトアウト → タイトルロゴ（ゲーム名は3人の入力語から自動合成。例:「ネオン・スライム・カーニバル」）がドーンと表示 |
| 音 | ビルド中はライザー（上昇音）、100%の瞬間に無音0.5秒 → BGM（Aが決めたジャンル）がフルボリュームで開幕 |
| 参加者の感情 | 期待のピーク。**無音0.5秒**が「息を呑む」を物理的に作る |

#### ◆ Phase 4: 完璧（7:30 – 9:00）「私たちのゲームが、動いている」

| 要素 | 設計 |
|---|---|
| PC画面 | 完成したパーティアクションゲームが起動。ホストがデモプレイ（or 自動デモ走行）。Bの敵が定義通りに跳ね、Cの紙吹雪が舞い、AのBGMテンポに敵の動きが同期 |
| 仕掛け | 画面下部に常時クレジット: `WORLD by ○○ / ENEMIES by ○○ / FX by ○○`。クリア時にCが定義したフィナーレ演出 + 3人の名前がスタッフロール風に流れる |
| スマホ側 | 観戦モード。「いいね」ボタン連打で画面に絵文字が飛ぶ（手持ち無沙汰の解消 + 熱狂の増幅） |
| 参加者の感情 | **熱狂と誇り**。「自分の言葉がゲームのルールとして動いている」という確信 |

#### ◆ Phase 5: 解説（9:00 – 10:00）

→ セクション2「学びの定着スクリプト」参照。感情の余熱が残るうちにパラダイムシフトを打ち込む。

## 3. 本番環境用アーキテクチャ設計

### 3.1 技術スタック（推奨構成）

| レイヤー | 技術 | 選定理由 |
|---|---|---|
| ホスト画面（ゲーム本体） | **Phaser 3** + TypeScript + Vite | 2Dアクションの定番。パラメータ駆動の物理・アニメ・パーティクルが揃う |
| 参加者スマホUI | 素のTS + CSS（or Preact） | 入力フォーム+ドラッグ配置のみ。軽量第一、インストール不要のWebアプリ |
| リアルタイム通信 | **Cloudflare Workers + Durable Objects（WebSocket）** | 1ルーム=1 DO で状態管理が素直。既存プロジェクト（commentar）で実績のあるスタックを流用 |
| ホスティング | Cloudflare Pages | 静的アセット配信 + Workers連携 + `npm run deploy` の既存運用に一致 |
| 言葉→パラメータ翻訳 | **Claude API**（`claude-haiku-4-5`、tool useによるJSON強制出力） | 低レイテンシ（目標2〜3秒）。スキーマ検証付き構造化出力で堅牢性確保 |
| 音楽 | パラメータ化済みBGMセット（ジャンル×テンポのマトリクス事前用意）+ Web Audio APIで再生レート微調整 | リアルタイム生成は不要。`bpm` パラメータで再生速度とゲーム内テンポを同期 |

### 3.2 システム構成図

```
[参加者スマホ ×3]                       [Cloudflare]                        [ホストPC]
┌──────────────┐                ┌─────────────────────┐              ┌────────────────────┐
│ 入力UI (PWA)  │── WebSocket ──▶│  Durable Object      │◀─ WebSocket ─│ ① ゲーム画面        │
│ ・お題フォーム │                │  (Room) 状態の正本    │              │   Phaser 3 / 投影用  │
│ ・スタンプ配置 │◀── 進行指示 ──│  ・フェーズ管理        │── params ───▶│ ② ホスト操作パネル   │
│ ・観戦/いいね │                │  ・入力キュー          │   patch      │   進行制御+カンペ    │
└──────────────┘                │  ・StageParams正本    │              └────────────────────┘
                                └──────┬──────────────┘
                                       │ 入力テキスト
                                       ▼
                                ┌─────────────────────┐
                                │ Worker: /translate    │
                                │ Claude API (tool use) │
                                │ 言葉 → StageParams片  │
                                │ + フォールバック辞書   │
                                └─────────────────────┘
```

### 3.3 データフロー（1入力のライフサイクル）

1. **入力**: スマホが `{type: "input_text", role: "B", prompt: "敵の見た目", text: "跳ね回るスライムドリンク"}` をWS送信
2. **即時ACK**: DOが受理を全クライアントへブロードキャスト → PC画面は即座に「光の粒が飛来」演出を開始（**体感レイテンシ0**）
3. **翻訳**: DOが翻訳Workerを呼び出し。Claude APIにロール別システムプロンプト + JSONスキーマ（tool use）で問い合わせ
4. **検証**: 返却JSONをZodスキーマで検証。範囲外の値はクランプ。失敗時はキーワード辞書フォールバック（3.5参照）
5. **反映**: DOが `StageParams` 正本にパッチをマージし、`{type: "params_patch", patch: {...}, sourceText: "...", author: "○○"}` を配信
6. **演出**: PC側Phaserが「翻訳ログ表示 → マテリアライズ演出 → 名前タグ」を再生。スマホへ `reflected` を返し振動+スタンプ演出

**スタンプ配置**はAI翻訳不要のため、座標正規化（0..1）のみで即時反映（目標 <300ms）。

### 3.4 StageParams スキーマ（ゲームの全状態 = この1オブジェクト）

```typescript
interface StageParams {
  environment: {
    palette: "neon" | "pastel" | "midnight" | "tropical" | "retro" | "monochrome";
    backdrop: "disco" | "rooftop" | "beach" | "space" | "kitchen" | "street";
    backdropFx: { fog: number; stars: number; lasers: number };      // 0..1
    bgm: { genre: "edm" | "funk" | "jazz" | "rock" | "lofi" | "8bit"; bpm: number }; // 80..180
    scrollSpeed: number;   // 0.5..2.0  ゲーム全体の速度係数
    gravity: number;       // 600..1400
  };
  enemies: Array<{
    id: string;
    archetype: "blob" | "spinner" | "chaser" | "floater" | "shooter"; // 挙動の型
    skin: { base: "slime" | "ball" | "ghost" | "bottle" | "cake"; tint: string; scale: number };
    behavior: { moveSpeed: number; bounce: number; jumpPower: number; aggression: number };
    spawn: { x: number; y: number } | "auto";  // スタンプ配置 or 自動
  }>;
  gimmicks: Array<{
    id: string;
    kind: "mirrorball" | "springpad" | "confettiCannon" | "slipFloor" | "spotlight";
    pos: { x: number; y: number };           // ミニマップ正規化座標
    params: Record<string, number>;           // kind別パラメータ
  }>;
  ui: {
    theme: "popart" | "cyber" | "handdrawn" | "luxury" | "kawaii";
    accentColor: string;
    particles: { type: "confetti" | "sparkle" | "bubbles" | "petals" | "pixels"; density: number };
    finale: { style: "fireworks" | "discoDrop" | "rainbowWave"; intensity: number };
  };
  meta: { title: string; credits: { role: "A" | "B" | "C"; name: string }[] };
}
```

> **アセット戦略:** 画像のリアルタイム生成は行わない（遅延・品質リスク）。`archetype × skin.base × tint` の**組み合わせ爆発で見た目の多様性を担保**する事前制作アセットマトリクス方式。スプライトは白ベースで描き、tintで着色。これにより翻訳は「選択+数値」の問題になり、AIの出力が100%安全にレンダリング可能になる。

### 3.5 AI翻訳レイヤー仕様

- **モデル:** `claude-haiku-4-5`（レイテンシ優先。表現力が不足する場合のみ `claude-sonnet-4-6` に切替）
- **方式:** tool use で `set_stage_params` ツールを強制呼び出し → スキーマ準拠JSONのみ返る
- **システムプロンプト要点:** 「あなたはゲームデザインの翻訳者。入力者の意図の**温度感**（激しさ・可愛さ・速さ）をパラメータの大小に反映せよ。スキーマ外の値は禁止。`translationLog` フィールドに『どの言葉をどのパラメータに翻訳したか』を1行ずつ日本語で出力せよ」
- **translationLog がビルド演出とホストのカンペの原料になる**（体験と学びの中核データ）
- **フォールバック:** API失敗/タイムアウト（4秒）時はキーワード辞書（「ネオン→neon」「速い→scrollSpeed 1.6」等、約100語）で即時変換。体験は途切れない
- **会場ネットワーク対策:** 全システムをNode.js（`ws` + ローカル静的配信）で動くローカルモード同梱。ホストPCでサーバを起動し、スマホは同一Wi-Fi + QR（ローカルIP）で接続。**本番はクラウド、保険にローカル**の二段構え

### 3.6 ホスト操作パネル（別ウィンドウ/別タブ）

- フェーズ進行ボタン（待機→接続→体現→ビルド→プレイ→解説）と各フェーズ残り時間
- 参加者ごとの入力ログ + 生成パラメータの対応表（**台本2.3のカンペ**）
- 「翻訳の全記録」画面（2.2）への切替ボタン
- 緊急ボタン: 入力スキップ / パラメータ手動修正 / デモ自動走行ON
